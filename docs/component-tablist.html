Log

  • 2020-12: tablist auto-activation should be enabled by default accodring to ARIA BP. So when a user keys to the next tab in the tablist, that Tab should auto-activate the tab panel. This is just BROKEN because in the ARIA spec the behavior is to alow keying through the tablist and on activating a tab the related tabpanel should show and be focussed. The user would be jumping around chaotically with this. E.g. user keys right on the tablist, next tab becomes active and the moves focus to the related tabpanel. The user can no longer key right to go to the next tab. S/he would have to shift+tab to back to the tablist and then key right. No UIs (as far as I know) do something like this. Recommend not doing, or making auto-activation a feature/flag that is disabled by default.
  • 2020-12: aria-orientation (on role=tablist) set to horizontal/vertical depending on layout. Is a property so you can update it dynamically, say from horizontal to vertical as a screen size shrinks for mobile. BUT what value does this offer? Who benefits from these semantics - not Screen reader users I would think? Also, this removes consistency with all other list navigation that uses up/left and down/right for navigation. Then there is perf concerns about adding a JS listener on resolution changes. NOTE: role=tablist has an implicit aria-orientation=horizontal.
